# 妙鉴 Prompt 交付质量检查清单

## 输入链路判断

- [ ] 是否判断清楚用户提供的是规则文档、正反例，还是两者都有。
- [ ] 有规则文档时，是否以规则文档为主。
- [ ] 无规则文档时，是否先从正反例归纳规则，再生成 prompt。
- [ ] 材料不足时，是否向用户补问必要信息。

## 能力边界

- [ ] 是否没有把示例场景当作 skill 的能力边界。
- [ ] 是否没有为每个电商场景单独生成或维护模板。
- [ ] 是否使用统一模板适配当前巡检任务。
- [ ] 是否可以迁移到其他电商巡检场景。

## 规则文档链路检查

- [ ] 是否覆盖规则文档中的全部核心标签、维度或规则组。
- [ ] 是否覆盖关键子规则和具体命中条件。
- [ ] 是否把细粒度规则转写为可执行的识别点。
- [ ] 是否保留正例、反例、白名单、豁免。
- [ ] 是否明确规则优先级，如首坑优先、A AND B、任一命中、全过程判断等。
- [ ] 是否明确多规则冲突时的输出策略。

## 正反例归纳链路检查

- [ ] 是否整理了正例和反例的样本类型、输入模态、人工标签、判定原因和业务备注。
- [ ] 是否对比了正反例的稳定差异，而不是只解释单个样本。
- [ ] 是否抽象出可判断的通用维度。
- [ ] 是否形成了定义、命中条件、排除条件、正向表现、负向表现、边界说明。
- [ ] 是否避免把样本中的具体商品名、文案、query、画面直接当成规则。
- [ ] 是否处理了样本不足、边界冲突或证据不足的不确定情况。

## 多模态适配

- [ ] 是否识别实际输入模态：文本、图片、视频、音频、截图、结构化字段等。
- [ ] 是否没有假设输入只有图片或只有文字。
- [ ] 是否为不同模态设置对应证据要求。
- [ ] 是否明确 OCR、ASR、MMU、意图识别等外部模型结果的使用方式。

## 结构完整

- [ ] 是否包含 `# 角色`。
- [ ] 是否包含 `# 背景说明`。
- [ ] 是否包含 `# 巡检任务`。
- [ ] 是否包含 `# 输入信息`。
- [ ] 是否包含 `# 规则来源`。
- [ ] 是否包含 `# 规则定义`。
- [ ] 是否包含 `# 判定原则`。
- [ ] 是否包含 `# 质检执行步骤`。
- [ ] 是否包含 `# 输出要求`。

## 输出可解析

- [ ] JSON 字段名稳定且符合业务含义。
- [ ] 布尔值使用 `true/false`，不是字符串。
- [ ] 空值使用 `null`。
- [ ] JSON 示例无注释、无多余逗号。
- [ ] 字段值枚举清晰，如 `"yes"/"no"`。
- [ ] 数组字段和字符串字段区分明确。
- [ ] 如果规则要求自然语言输出，是否没有强行改成 JSON。

## 判定清晰

- [ ] 正常与异常边界明确。
- [ ] 命中条件和豁免条件不冲突。
- [ ] 对未知、不确定或证据不足情况有处理说明。
- [ ] 对内部判定依据和最终输出标签有明确区分。
- [ ] 有 `evidence`、`text_evidence`、`visual_evidence`、`audio_evidence`、`video_evidence` 或 `reason` 支撑判定。

## 平台适配

- [ ] 输入占位符明确，如 `{content}`、`{image}`、`{video}`、`{audio}`、`{ocr_text}`、`{asr_text}`、`{structured_fields}`、`{positive_examples}`、`{negative_examples}`、`{model_result}`。
- [ ] 输出字段满足妙鉴平台下游解析需求。
- [ ] prompt 语言风格与既有示例一致。
- [ ] 没有把规则文档中的保密页眉页脚写入 prompt 主体。
