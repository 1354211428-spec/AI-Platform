# 规则文档抽取指南

## 适用场景

当用户提供规则文档时，使用本指南。若用户同时提供正反例，仍以规则文档为主，正反例用于校准边界、补充反例、发现规则遗漏或冲突。

## 1. 抽取巡检目标

不要先按场景归类，而要先回答：这份规则到底要巡检什么？

常见目标包括但不限于：

- 判断是否异常、违规、不合规。
- 判断是否命中一个或多个标签。
- 判断是否符合正常白名单。
- 判断是否满足体验、相关性、准确性、承接性、可用性等要求。
- 判断是否与 MMU、OCR、ASR、意图模型或其他模型结果一致。

## 2. 抽取巡检对象与输入模态

识别规则作用于哪些对象和证据来源：

- 文本：评论、回复、标题、商品描述、页面文案、OCR 文本、ASR 文本等。
- 图片：商品图、封面图、页面截图、直播截图、素材图等。
- 视频：视频画面、关键帧、动作、字幕、封面、时序变化等。
- 音频：语音内容、ASR、语气、背景音等。
- 结构化字段：query、商品类目、用户画像、历史行为、模型标签、置信度、系统识别结果等。

在 prompt 中必须把这些输入写成清晰的占位符，不要只写「待检内容」。

## 3. 抽取规则层级

对规则建立层级，但不要预设只有一级/二级/三级：

```text
巡检目标
├── 规则组/一级标签/一级维度
│   ├── 子规则/二级标签/二级维度
│   │   ├── 具体命中条件/三级标准
```

然后决定输出层级：

- 规则写「只输出一级标签」：输出 JSON 只保留一级标签。
- 规则写「二级/三级只是判定标准」：二级/三级写入规则定义，不放入最终输出字段。
- 规则给出固定输出文案：严格使用规则指定文本。
- 规则要求自然语言判断：保留自然语言输出，不强行改为 0/1。

## 4. 抽取硬条件

以下表达必须显式写成强约束：

- 「命中任意一项」→ 任一条件成立即判定。
- 「必须同时满足」→ A AND B。
- 「优先判断」→ 该规则先于其他规则。
- 「首坑」「首屏」「全过程」「当前帧」「整段视频」→ 明确判定范围。
- 「白名单/豁免」→ 命中后不判异常或降低优先级。
- 「不输出 0/1」→ 不要强行写 `is_anomaly`，除非用户另有要求。

## 5. 抽取证据要求

根据输入模态设置证据字段：

- 文本类规则：要求输出 `text_evidence` 或 `evidence`。
- 图片/截图类规则：要求输出 `visual_evidence`。
- 视频类规则：要求输出 `video_evidence`，可包含关键帧、时间段、画面变化。
- 音频类规则：要求输出 `audio_evidence`，可引用 ASR 文本或声音特征。
- 结构化字段规则：要求指出命中的字段和值。

## 6. 抽取正例、反例与豁免

- 正例用于帮助模型理解正常边界。
- 反例用于帮助模型识别 Badcase。
- 豁免必须靠近对应规则，避免误伤。
- 如果反例只是参考库，需要写明「实际以是否命中核心规则为准」。
- 如果用户额外提供示例，检查示例是否与规则文档一致；如冲突，默认以规则文档为准，并提示该冲突可作为后续规则修订点。

## 7. 抽取输出要求

优先从规则文档找明确要求：

- 0/1。
- true/false。
- 固定文本标签。
- JSON。
- 自然语言原因。
- 多模态证据字段。
- 模型一致性字段。

若规则未给输出格式，默认使用 JSON，并至少包含：

```json
{
  "is_anomaly": true,
  "anomaly_tags": "命中的标签或无异常",
  "matched_rule": "命中的规则名称或编号",
  "evidence": "支撑判定的关键证据",
  "reason": "判定原因",
  "confidence_score": 0.0
}
```

## 8. 外部模型一致性抽取

如果输入包含 `mmu_type`、`ocr_result`、`asr_result`、`intent_result`、`model_result`、`系统识别结果`，追加一致性字段：

- AI 巡检结果与入参模型结果一致 → `is_consistent_with_model = "yes"`
- 不一致 → `is_consistent_with_model = "no"`

必须明确每种标签或结果的对应关系，避免只比较字面文本。
