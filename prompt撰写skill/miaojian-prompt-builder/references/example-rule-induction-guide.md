# 正反例规则归纳指南

## 适用场景

当用户没有规则文档，只提供正面例子和反面例子时，使用本指南。目标不是复述样本，而是从样本中抽象出可迁移、可执行、可用于妙鉴巡检 prompt 的规则。

## 1. 整理样本

先把用户给的例子整理为结构化信息：

```text
样本编号
├── 样本类型：正例 / 反例
├── 输入模态：文本 / 图片 / 视频 / 音频 / 截图 / 结构化字段 / 多模态组合
├── 样本内容：原文、图片描述、视频关键帧、音频 ASR、字段值等
├── 人工标签：如有
├── 判定原因：如有
├── 业务备注：如有
```

如果样本缺少判定原因，需要从正反例差异中推断，但不要过度臆测。

## 2. 对比正反例差异

从以下维度寻找稳定差异：

- **语义维度**：是否相关、是否答非所问、是否夸大、是否承诺过度、是否误导。
- **视觉维度**：主体、背景、文字、布局、颜色、动作、位置、画面质量。
- **视频维度**：关键帧、时序变化、动作过程、字幕、封面与内容一致性。
- **音频维度**：ASR 文本、语义、语气、背景音、是否与画面或文案一致。
- **行为/交互维度**：点击诱导、承接路径、用户意图满足、操作可用性。
- **商品/电商维度**：商品类目、价格、促销、承接卡片、购买入口、商品与内容一致性。
- **体验维度**：是否造成困惑、不适、低质、单调、误解或不信任。

优先寻找能稳定区分多数正反例的共性，而不是只解释单个样本的偶然特征。

## 3. 抽象判定维度

将差异归纳为可判断的维度，例如：

- 内容是否与用户意图相关。
- 是否存在违规、夸大或误导表达。
- 是否具备目标承接能力。
- 是否存在视觉/语音/视频层面的异常信号。
- 是否符合平台体验标准。
- 是否与商品、场景或上下文一致。

每个维度都要能被后续模型通过输入证据判断。

## 4. 形成规则

为每个归纳出的维度写成规则，推荐结构：

```markdown
## 规则：{规则名}
定义：{这条规则判断什么}
命中条件：
- {条件1}
- {条件2}
排除条件/白名单：
- {什么情况下不算问题}
正向表现：
- {从正例中抽象出的通过特征}
负向表现：
- {从反例中抽象出的异常特征}
边界说明：
- {容易误判的灰区和处理方式}
证据要求：
- {需要文本/视觉/音频/视频/结构化字段中的哪些证据}
```

## 5. 泛化规则，避免过拟合

从正反例归纳规则时，必须把样本细节抽象为通用标准：

- 不要只写具体商品名、query、主播名、页面文案；要抽象为「商品与意图不一致」「承接入口缺失」「回复未回答用户问题」等规则。
- 不要只描述某一张图或某一句话；要抽象为可在新样本中识别的视觉、语义或行为特征。
- 不要把所有反例特征都写成必要条件；区分「核心条件」和「辅助信号」。
- 不要把正例的偶然特征写成白名单；白名单必须有明确业务理由。

## 6. 处理样本不足和冲突

如果正反例数量少、边界冲突或信息不足：

- 生成保守规则，不要过度扩展。
- 在规则中加入「不确定情况处理」。
- 需要时向用户补问：
  - 这些正反例的人工判定原因是什么？
  - 是否有边界样本？
  - 最终输出需要哪些标签？
  - 是否宁可误杀还是漏放？

可在 prompt 中写：

```markdown
如果输入证据不足以判断是否命中规则，不要臆测；应输出低置信度，并在 reason 中说明缺少哪些证据。
```

## 7. 归纳结果填入统一模板

将归纳结果填入：

- `{source_type}`：正反例归纳
- `{rule_definitions}`：归纳出的规则定义、命中条件、排除条件
- `{example_boundary_notes}`：正反例边界、灰区、不确定情况
- `{decision_principles}`：规则优先级、冲突处理、证据不足处理
- `{output_schema}`：按用户或平台要求生成
